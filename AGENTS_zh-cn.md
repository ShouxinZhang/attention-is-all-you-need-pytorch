# 仓库代理指令

- 代码应尽可能遵循模块化设计原则；不同模块应避免相互干扰。任何实验性功能都应将其所有文件放置在单个子模块中，且不得在较大的模块中生成冗余文件。
- 在编写新代码之前，请查阅现有代码以减少冗余。
- 任何新更改应保持尽可能简单。请勿添加额外的分支或未经授权的功能规划。
- 在任何实现中，业务成果第一。首要任务是交付所需的功能，其次是长期架构规划，然后是保持模块风格简洁高效。
- 严禁将任何新代码存储在高级模块中（例如，仓库根目录或其他高级模块目录）。必须根据实际使用情况将其放置在叶级模块目录中。
- docs/architecture/repository-structure.md 是仓库架构文件。在每次更改代码之前，必须查阅此文件以确保更改遵循仓库架构。修改后，立即更新此文件以保持与仓库的一致性。

向用户解释代码时，始终从业务角度进行描述。即使用户是开发人员，也将其视为仅想知道代码能为业务做什么、其优点和缺点的管理领导者。

在执行任何代码更改之前，必须与用户达成共识。只有在用户确认方案可行后才开始工作。

在任何文件删除或回退之前，必须创建备份。

正面提示词（规范操作，非常鼓励的行为):

- 前端代码写完后, 自行check lint以及npm run dev，查看是否有报错, 如果有错误，就自行修改, 直到没有错误为止
- docs/dev_logs中存放每一次开发的日志, 需要详细记录修改了哪些文件, 做了什么. 每次修改完都应该更新dev_logs. dev_logs文档最好是可以类似于数据库的特性那样，可以随时回滚快照，储存格式为开发日期为文件夹，文件夹内文件名称为开发内容，文件内需要显示修改的具体内容，修改了哪些文件，以及修改时间（精确到秒)
- 自动更新repository-structure.md
- 凡是可以帮助用户自动化完成的，一律帮助用户自动化完成，而不是要求用户自己操作。（如果需要用户提供必要信息，请提示)
- 由于你的知识很可能是陈旧的，因此当涉及到安装SDK或者py package服务等, 请你在线获取最新版本后，默认安装最新版本，而不是某个落后的版本号

负面提示词（严厉禁止的行为):

- 写完代码不作任何检查, 也不自主运行测试, 很多报错和bug
- 开发完成不在dev_logs里写日志，造成开发记忆丢失, 后期维护困难
- 开发新的功能后，不写repository-structure.md文档
- 偷懒，把你自己就可以在命令行或者写一个脚本就完成的任务，让用户自己麻烦的动手操作（反面例子：你（用户）)把xxx完成，我（LLM)就可以xxx，这是绝对被禁止的！如果你不能独立完成，你必须说明，提供xxx信息后，我可以xxx)
- 安装某个落后的版本SDK或者package, 例如mathlib 4.26, 或者pytorch 2.7之类，最新版本早已超出了这些旧的版本，进而给用户带来版本问题
